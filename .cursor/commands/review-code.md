# 🔍 程式碼審查

像索隆檢查他的刀一樣仔細審查這段程式碼！

## 審查項目

### 1. 🎯 功能正確性
- [ ] 程式碼是否符合需求？
- [ ] 邏輯是否正確？
- [ ] 邊界條件處理？
- [ ] 錯誤處理完整？

### 2. 🏗️ 架構設計
- [ ] 符合 SOLID 原則？
- [ ] 適當的設計模式？
- [ ] 模組職責清晰？
- [ ] 耦合度是否過高？

### 3. 💅 程式碼品質
- [ ] 命名是否清晰？
- [ ] 函數是否過長？（建議 < 50 行）
- [ ] 複雜度是否過高？（圈複雜度 < 10）
- [ ] 重複程式碼？

### 4. ⚡ 效能考量
- [ ] 有無效能瓶頸？
- [ ] 資料庫查詢優化？
- [ ] 記憶體使用合理？
- [ ] 有無不必要的運算？

### 5. 🔐 安全檢查
- [ ] 輸入驗證完整？
- [ ] SQL 注入防護？
- [ ] XSS 防護？
- [ ] 敏感資料處理？
- [ ] 權限檢查？

### 6. 🧪 測試覆蓋
- [ ] 單元測試存在？
- [ ] 測試覆蓋率足夠？
- [ ] 邊界條件測試？
- [ ] 錯誤路徑測試？

### 7. 📚 文檔完整性
- [ ] 函數註解清楚？
- [ ] 複雜邏輯有說明？
- [ ] API 文檔更新？
- [ ] README 更新？

## 輸出格式

### 🟢 優點
列出做得好的地方

### 🔴 必須修改
列出必須修改的問題（阻塞合併）

### 🟡 建議改進
列出建議改進的地方（非阻塞）

### 💡 最佳實踐建議
提供改進的具體程式碼範例

### 📊 評分
- 功能性: X/10
- 可讀性: X/10
- 維護性: X/10
- 效能: X/10
- 安全性: X/10

總評: X/50
